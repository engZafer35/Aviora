Protocol Layer Overview

The Aviora offers several interchangeable protocol stacks in source form: JSON or binary TLV over TCP, and a JSON session model over MQTT. They share the same meter and job pipeline through AppMeterOperations, while transport is handled by AppTcpConnManager (TCP) or AppMqttConnService (MQTT). For a given customer or product build, you enable one stack—for example Orion TLV for one deployment and RigelMq for another—by wiring the right module at integration time; switching backends does not require rewriting unrelated subsystems.

Customer Builds and Compile-Time Selection

Aviora already uses per-customer configuration (JSON and generated headers) for file system, logging, time service, and similar areas. The protocol layer follows the same idea: the firmware image includes exactly one active protocol service—whichever implementation your integration layer links and starts (appProtocol*Init / appProtocol*Start). One customer image might call Orion TLV + TCP only; another might call RigelMq + MQTT only. You do not run all parsers at once in production; you select the right service for that binary.

Typical split. Customer A’s head-end expects binary TLV with transaction ids → build with AppProtocolOrionTLV. Customer B’s cloud uses MQTT topics and JSON sessions → build with AppProtocolRigelMq. Meter logic, directives, and readout jobs stay behind the same AppMeterOperations API; only the wire codec and transport task change.

Isolated Protocol Service

The protocol unit is designed as a self-contained service: it owns framing, sessions, and callbacks into the transport layer, and talks to the rest of the gateway through narrow interfaces (incoming message handlers, meter operations, optional file or timer hooks). Display, UI, unrelated drivers, and other services do not need to know which wire format is active—only that metering and jobs are requested consistently.

What You Configure

Alongside choosing the module, you supply endpoints from customer config: server IP and ports for TCP stacks, broker and topics for MQTT, serial number, pull port, credentials, and so on—usually fed from the same customer JSON or generated constants used elsewhere in the port.

Push vs Pull (TCP-based Stacks)

Channel Who initiates Typical use
Push Device connects to push socket of the server via TCP ident, alive, readout / load-profile data toward the backend
Pull Device listens locally; server connects in log, setting, fwUpdate, readout requests, directive list/add/delete

Details: TCP Connection Service — non-blocking select, push and pull sockets, request-based connect/disconnect.
